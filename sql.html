Basic tables View lesson

In SQL, data is stored in tables made up of columns, similar to a spreadsheet. We can CREATE tables, INSERT data into them, and SELECT the data back out.

>
CREATE TABLE users (email TEXT, name TEXT);
INSERT INTO users (email, name)
  VALUES ('amir@example.com', 'Amir');
SELECT * FROM users;
Result:
[{email: 'amir@example.com', name: 'Amir'}]

No type enforcement in SQLite View lesson

Most SQL databases enforce column types, but SQLite doesn't. It will let us insert the wrong type of data into a column.

>
CREATE TABLE rects (width REAL, height REAL);
INSERT INTO rects (width, height)
  VALUES ('oh', 'no');

Null View lesson

SQL has NULL, which is similar to the nulls in other languages like JavaScript. If we declare a column as NOT NULL, the database won't allow us to insert nulls.

>
CREATE TABLE users (name TEXT NOT NULL);
INSERT INTO users (name) VALUES (NULL);
Result:
Error: NOT NULL constraint failed: users.name
Selecting columns View lesson

We can select specific columns by naming them in the SELECT statement.

>
CREATE TABLE users (name TEXT, logins INTEGER);
INSERT INTO users (name, logins)
  VALUES ('Amir', 1);

Result:
[{name: 'Amir'}]

UPDATE modifies rows that already exist.

  WHERE name = 'Keanu';
Result:

If we UPDATE without a WHERE, we'll update all rows. Be careful not to do this accidentally!

[{name: 'Cat'}, {name: 'Cat'}]
Unique constraints View lesson

A unique constraint ensures that no two rows have the same value in a column or set of columns. For example, we might want to ensure that each pair of user name and cat name is unique.

This constraint is on two columns, name and cat_name. There can be duplicate names, and duplicate cat_names. But no row can have the same name and cat_name as another row.

>
CREATE TABLE users (
  name TEXT NOT NULL,
  cat_name TEXT NOT NULL,
  UNIQUE (name, cat_name));
INSERT INTO users (name, cat_name)
  VALUES ('Amir', 'Ms. Fluff'),
  VALUES ('Amir', 'Ms. Fluff');
Result:
Error: UNIQUE constraint failed: users.name, users.cat_name

Primary keys View lesson

Primary keys are normally unique identifiers for rows. We often use integers, but sometimes we use special strings like UUIDs.

>
CREATE TABLE people (
  id INTEGER PRIMARY KEY,
  name TEXT);
Result:

Primary keys must be unique.

>
CREATE TABLE people (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL);
INSERT INTO people (id, name)
  VALUES (1, 'Amir'),
  VALUES (1, 'Betty');

Error: UNIQUE constraint failed: people.id

Foreign keys View lesson

When a table references columns from another table, we can configure the database to ensure that referenced records actually exist. For example, we'd like the database to only accept a cat with an owner_id of 3 if a user with the ID of 3 exists.

>
CREATE TABLE people (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL);
/* Every cat row's owner_id must reference the
 * ID of a person row. */
CREATE TABLE cats (
  owner_id INTEGER REFERENCES people(id),
  name TEXT NOT NULL);
Result:
Comparing with null View lesson

Comparisons and mathematical operations on NULL return another NULL.

>
SELECT NULL = 1 AS result;
Result:
[{result: null}]
>
SELECT NULL + 1 AS result;
Result:
[{result: null}]

Even comparing NULL = NULL gives us another NULL back.

>
SELECT NULL = NULL AS result;
Result:
[{result: null}]

If we need to check for whether a value is actually NULL, we can use the IS NULL and IS NOT NULL operators.

>
SELECT 5 IS NOT NULL AS result;
Result:
[{result: 1}]

Uniqueness constraints normally disallow duplicate values in the column. However, duplicate NULLs are always allowed.

>

  VALUES (NULL),

Constraint analysis View lesson

There are three important questions to ask for every column:

  • Should this column be a foreign key?
  • Can this column be null?
  • Can multiple rows have the same value for this column?
Simple joins View lesson

Joins let us query multiple tables that relate to each other, whether via a foreign key or not.

>
/* Select all cats' names along with their
 * owners' names. */
SELECT
  people.name AS person,
  cats.name AS cat
FROM people
JOIN cats
  ON people.id = cats.owner_id;
Result:
Join performance View lesson

The easy way to think about a join is: "it creates all possible combinations of rows from both tables, then filters them using the ON clause". However, the database engine doesn't run them that way. Its query optimizer will run queries in microseconds when they'd otherwise take years.

Left and right joins View lesson

Left joins always include rows from the left table, even if the ON clause didn't match for those rows. Right joins always include rows from the right table.

>
/* Select users and their discounts. Users with
 * no discounts will still show up, but the
 * discount columns will be null. */
SELECT *
FROM users LEFT JOIN discounts
  ON users.discount_id =
  discounts.id;
Result:
On conflict do nothing View lesson

We can tell the database what to do when we insert records that conflict with constraints. The simplest form is ON CONFLICT DO NOTHING, which tells the database to ignore the insert.

>
CREATE TABLE vaccinations (
  cat_name TEXT UNIQUE NOT NULL);
INSERT INTO vaccinations (cat_name)
  VALUES ('Ms. Fluff');
INSERT INTO vaccinations (cat_name)
  VALUES ('Ms. Fluff')
  ON CONFLICT (cat_name) DO NOTHING;
SELECT * FROM vaccinations;
Result:
[{cat_name: 'Ms. Fluff'}]
Selecting expressions from tables View lesson

When SELECTing from a table, we can SELECT expressions computed from the table's columns.

>
CREATE TABLE cats (name TEXT, age INTEGER);
INSERT INTO cats (name, age)
  VALUES ('Ms. Fluff', 3);
SELECT age, age + 1 AS age_next_year FROM cats;
Result:
[{age: 3, age_next_year: 4}]
SQL injection View lesson

When working with SQL in another language, never concatenate SQL strings directly like sql1 + sql2! It opens the possibility of attackers crafting special values that let them execute arbitrary SQL code.

On conflict update View lesson

With ON CONFLICT UPDATE, a single INSERT statement can either insert a record or update an existing record. The ON CONFLICT UPDATE clause is only triggered when a constraint is violated.

>
/* If Amir exists, increment the count on his
 * existing row by 1. Otherwise, insert a row
 * for Amir. */
INSERT INTO visits (email, count)
  VALUES ('amir@example.com', 1)
  ON CONFLICT (email)
    DO UPDATE SET count = count + 1;

Join mistakes View lesson

Here are two important tips for testing your joins:

  • When testing a join, use more than one row in each table.
  • When selecting from a join, qualify your selects with table names (like SELECT users.name) to make sure that you're selecting the right things.
>
/* This join is missing its "ON", so it returns
 * all combinations of users and comments. We
 * may not notice this if we only test simple
 * cases. */
SELECT users.name, comments.text
FROM users JOIN comments

ON vs WHERE View lesson

It's possible to get the same INNER JOIN results using ON or WHERE. Try to use the one that communicates your intent to other programmers. If your condition concerns which rows are joined with other rows, use ON. If you think of your condition as filtering the result of the join, use WHERE.

However, ON and WHERE are not equivalent for left, right, or outer joins!

>
/* We use an ON here because our conditions
 * concern which rows are joined with other
 * rows. */
SELECT
  people.name AS person_name,
  cats.name AS cat_name
FROM people
INNER JOIN cats
  ON people.id = cats.owner_id
  AND people.name = cats.name
